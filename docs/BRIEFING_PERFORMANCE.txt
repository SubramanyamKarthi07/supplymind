2. ROOT CAUSE ANALYSIS
===========================================================

Performance Bottleneck Identified

Database Reads

Cause

Every Executive Briefing request loaded supplier,
supplier performance and inventory data directly
from PostgreSQL.

Operations Included

• Reading Suppliers Table
• Reading Supplier Performance Table
• Reading Inventory Table

Impact

Repeated database access introduced significant
latency for every request.

Root Cause

Database I/O was the primary performance bottleneck,
not intelligence processing.


===========================================================
3. OPTIMIZATIONS IMPLEMENTED
===========================================================

Optimization 1
Database Read Refactoring

Previous

Every request executed database queries.

Current

Database tables are loaded once and reused.


-----------------------------------------------------------

Optimization 2
Database Cache

Implemented DatabaseCache class.

Cached Objects

• Suppliers
• Supplier Performance
• Inventory Positions

Result

Subsequent requests reuse cached DataFrames instead
of querying the database again.


-----------------------------------------------------------

Optimization 3
Reusable Data Access

Replaced

pd.read_sql(...)

with

db_cache.get_suppliers()

db_cache.get_supplier_performance()

db_cache.get_inventory()

This eliminated repeated database reads.


-----------------------------------------------------------

Optimization 4
Parallel Intelligence Processing

Relationship Analysis

Competitive Analysis

Scenario Analysis

executed using ThreadPoolExecutor.

Although these functions were already fast,
parallel execution improves scalability for future
enhancements.
